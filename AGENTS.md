# AGENTS.md

## What this repo is

This is the GitHub **profile README** repository (`ruzickap/ruzickap`).
It contains only Markdown and tooling config — there is **no application
code, no build step, no tests, and no package manager**. Do not add a
`package.json`, Terraform, or a CI test job; they don't belong here.

`README.md` is what renders on the GitHub profile page.

## Do not hand-edit generated content

- The blog list in `README.md` between the
  `<!-- BLOG-POST-LIST:START -->` / `END` markers is auto-generated weekly
  by `.github/workflows/blog-post-workflow.yml` from
  `https://ruzickap.github.io/feed.xml`. Edit the markers/surrounding text,
  never the list entries.
- `CHANGELOG.md` is generated by release-please. It is excluded from every
  linter — leave it alone.

## Linting (no local runner is committed)

All checks run via **MegaLinter** (`documentation` flavor) in CI; there is
no `make`/`npm` target. Config lives in `.mega-linter.yml`. To reproduce a
single check locally, run the underlying tool directly with its repo config:

- Markdown: `rumdl check .` (config `.rumdl.toml`; `markdownlint` is
  intentionally disabled in favor of rumdl).
- Links: `lychee .` (config `lychee.toml` + `.lycheeignore`;
  `markdown-link-check` is disabled — ignore `.markdown-link-check.json`).
- Shell **inside** ```` ```bash ```` / `shell` / `sh` code fences is
  extracted and validated: `shellcheck --exclude=SC2317` and
  `shfmt --case-indent --indent 2 --space-redirects`. Keep fenced shell
  snippets valid and use uppercase `${VARIABLES}`.

CI runs MegaLinter only on **non-`main` branches** and only when a
`**/*.md` file changed, so push doc work on a branch, not to `main`.

## Commits, branches, PRs

- **Conventional Commits** with a scope, e.g. `docs(readme): ...`,
  `feat(gha): ...`, `chore(deps): ...`. Subject ≤ 72 chars, lowercase, no
  trailing period, imperative mood. Enforced by `commit-check` on PRs to
  `main`.
- **PR titles** must also be conventional (enforced by
  `semantic-pull-request`).
- **Branches** must follow Conventional Branch (`feature/`, `bugfix/`,
  `hotfix/`, `release/`, `chore/`); lowercase + hyphens only. Enforced by
  `commit-check` (bots are exempted).
- Releases are automated by **release-please** (`simple` type) on merge to
  `main` — do not bump versions or edit `CHANGELOG.md` manually.

## GitHub Actions conventions

When editing anything under `.github/workflows/`:

- Validate with `actionlint`.
- Pin every action to a **full commit SHA** (with a `# vX.Y.Z` comment),
  never a tag — existing workflows show the pattern.
- Keep `permissions: read-all` at the top level and grant write scopes only
  on the specific job that needs them.
- Workflows are also scanned by `checkov` (`.checkov.yml` skips
  `CKV_GHA_7`), `zizmor`, `trivy`, and `DevSkim` via MegaLinter.
